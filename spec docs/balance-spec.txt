# DIFFICULTY & BALANCING SYSTEM SPECIFICATION

## 1. PURPOSE

The Difficulty System replaces hardcoded progression logic with a **fully data-driven configuration layer**.

It must:

* Allow **rapid tuning without code changes**
* Centralise all gameplay balance variables
* Support **progression, scaling, and unlock pacing**
* Enable **late-stage playtesting adjustments without refactoring**

---

## 2. CORE PRINCIPLE

All gameplay-affecting values must come from a **single external configuration object**.

No gameplay logic should rely on:

* hardcoded thresholds
* inline constants
* scattered tuning values

---

## 3. CONFIG STRUCTURE

### 3.1 Root Object

```js
DifficultyConfig = {
  levels: LevelConfig[],
  scoring: ScoringConfig,
  timing: TimingConfig,
  unlocks: UnlockConfig,
  constraints: ConstraintConfig
}
```

---

## 4. LEVEL CONFIG

Defines progression tiers.

```js
LevelConfig = {
  level: number,
  label: string,

  progression: {
    scoreThreshold: number
  },

  gameplay: {
    maxChordDifficulty: number,
    chordPool: string[] | null
  },

  timing: {
    hintDurationMs: number,
    responseTimeSec: number | null
  },

  rewards: {
    baseScore: number,
    timeBonusMultiplier: number,
    streakMultiplier: number
  }
}
```

---

### 4.1 Notes

* `maxChordDifficulty` replaces current `diff` filtering
* `chordPool` allows explicit overrides (optional)
* `responseTimeSec = null` means no timer

---

## 5. SCORING CONFIG

Centralises all score calculations.

```js
ScoringConfig = {
  basePoints: number,
  timeBonus: {
    enabled: boolean,
    maxBonus: number,
    curve: "linear" | "easeOut"
  },
  streak: {
    increment: number,
    decayOnFail: boolean,
    maxMultiplier: number
  }
}
```

---

### 5.1 Example Calculation

```js
score =
  basePoints
  + (timeRemainingRatio * maxBonus)
  * (1 + streak * increment)
```

---

## 6. TIMING CONFIG

Global timing behaviours.

```js
TimingConfig = {
  hintFadeCurve: "linear" | "easeInOut",
  successDelayMs: number,
  failDelayMs: number,
  inputGraceMs: number
}
```

---

## 7. UNLOCK SYSTEM

Defines when systems/features become available.

```js
UnlockConfig = {
  levels: {
    shop: number,
    patching: number,
    modulation: number
  },

  modules: [
    {
      type: string,
      unlockLevel: number,
      maxInstances: number | null
    }
  ],

  parameters: [
    {
      param: string,
      unlockLevel: number
    }
  ]
}
```

---

## 8. CONSTRAINT SYSTEM (CRITICAL FOR GAME DESIGN)

Controls complexity at each stage.

```js
ConstraintConfig = {
  maxActiveNotes: number,
  maxModules: number,
  maxConnections: number,
  maxModulationDepth: number,

  allowedModuleTypes: string[],
  allowedConnectionTypes: ("audio" | "mod")[]
}
```

---

## 9. CHORD DIFFICULTY SYSTEM

Replace implicit difficulty with explicit mapping.

```js
ChordDifficulty = {
  "major": 1,
  "minor": 2,
  "dom7": 3,
  "maj7": 4,
  "min7": 4,
  "dim": 5
}
```

---

## 10. CHORD POOL GENERATION

The system must support two modes:

### 10.1 Procedural (default)

* Filter by:

  * key (if in play mode)
  * `maxChordDifficulty`

### 10.2 Explicit Pool

* Use `level.gameplay.chordPool` if provided

---

## 11. DIFFICULTY CURVES

The system must allow:

* Increasing chord complexity
* Decreasing hint duration
* Increasing scoring pressure
* Expanding available modules

All driven by config only.

---

## 12. DYNAMIC DIFFICULTY (OPTIONAL)

Support runtime adjustment:

```js
DynamicDifficulty = {
  enabled: boolean,
  adjustOn: {
    successStreak: number,
    failureRate: number
  },
  modifiers: {
    hintDurationScale: number,
    timeLimitScale: number
  }
}
```

---

## 13. GAME INTEGRATION

### 13.1 Level Resolution

```js
currentLevel = levels.findLast(l => score >= l.progression.scoreThreshold)
```

---

### 13.2 Runtime Usage

Game systems must query config instead of hardcoding:

* Hint timing → `level.timing.hintDurationMs`
* Allowed chords → `level.gameplay`
* Score → `scoring`
* Unlocks → `unlocks`

---

## 14. DESIGN WORKFLOW

This system enables:

* Designers tweak JSON → immediate effect
* No code changes required for:

  * balancing
  * pacing
  * progression tuning

---

## 15. EXAMPLE CONFIG

```js
{
  levels: [
    {
      level: 1,
      label: "LEVEL 1",
      progression: { scoreThreshold: 0 },
      gameplay: { maxChordDifficulty: 1, chordPool: null },
      timing: { hintDurationMs: 2200, responseTimeSec: 5 },
      rewards: { baseScore: 100, timeBonusMultiplier: 1, streakMultiplier: 0.15 }
    },
    {
      level: 5,
      label: "LEVEL 5",
      progression: { scoreThreshold: 2500 },
      gameplay: { maxChordDifficulty: 3, chordPool: null },
      timing: { hintDurationMs: 1000, responseTimeSec: 3 },
      rewards: { baseScore: 140, timeBonusMultiplier: 1.2, streakMultiplier: 0.2 }
    }
  ],

  scoring: {
    basePoints: 100,
    timeBonus: { enabled: true, maxBonus: 100, curve: "linear" },
    streak: { increment: 0.15, decayOnFail: true, maxMultiplier: 3 }
  },

  unlocks: {
    levels: { shop: 10, patching: 8, modulation: 12 },
    modules: [
      { type: "oscillator", unlockLevel: 1, maxInstances: 3 },
      { type: "filter", unlockLevel: 4, maxInstances: 2 }
    ],
    parameters: [
      { param: "lfoDepth", unlockLevel: 6 }
    ]
  },

  constraints: {
    maxActiveNotes: 6,
    maxModules: 8,
    maxConnections: 16,
    maxModulationDepth: 3,
    allowedModuleTypes: ["oscillator", "filter", "env", "output"],
    allowedConnectionTypes: ["audio"]
  }
}
```

## 16. DESIGN GOALS SUMMARY

This system must be:

* Fully data-driven
* Centralised
* Easily tunable
* Decoupled from logic
* Safe to modify late in development
